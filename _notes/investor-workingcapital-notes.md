## 1. `fineract-investor` — Loan Asset Externalization

### Purpose
The `fineract-investor` module enables **loan asset externalization** (securitization / loan sale). It allows a financial institution to transfer the economic ownership of one or more loans to an external investor ("external asset owner"), while the loan continues to be serviced internally. The module is **conditionally enabled** via `InvestorModuleIsEnabledCondition`.

### Key Domain Classes

#### `ExternalAssetOwner` (table: `m_external_asset_owner`)
Represents an external party (investor) identified by a unique `ExternalId`. Owners are lazily created on first use via `ExternalAssetOwnerHelper.findOrCreateId()`, which runs in a `REQUIRES_NEW` transaction to prevent constraint-violation corruption of the outer transaction.

#### `ExternalAssetOwnerTransfer` (table: `m_external_asset_owner_transfer`)
The central audit-trail entity for a loan ownership change event. Fields:
- `owner` – the target external investor
- `previousOwner` – the prior owner (for owner-to-owner re-sales)
- `externalId` – UUID-style identifier of this transfer
- `status` – see `ExternalTransferStatus` below
- `subStatus` – see `ExternalTransferSubStatus` below
- `purchasePriceRatio` – the ratio at which the loan is sold (e.g., "1.0" = par)
- `settlementDate` – the COB date on which the transfer is executed
- `effectiveDateFrom` / `effectiveDateTo` – the period for which this record is the authoritative view
- `loanId` / `externalLoanId` – the loan being transferred
- `externalGroupId` – optional grouping for batch transfers

#### `ExternalTransferStatus` (enum)
```
ACTIVE            – loan is currently owned by this investor
ACTIVE_INTERMEDIATE – intermediary/warehouse owner (delayed settlement path)
DECLINED          – transfer was rejected on settlement date (see sub-statuses)
PENDING           – sale requested, awaiting next COB cycle
PENDING_INTERMEDIATE – sale to intermediary, awaiting COB
BUYBACK           – buyback requested, awaiting next COB cycle
BUYBACK_INTERMEDIATE – buyback from intermediary, awaiting COB
CANCELLED         – transfer cancelled (e.g., same-day sale+buyback)
```

#### `ExternalTransferSubStatus` (enum)
```
BALANCE_ZERO      – declined because outstanding balance = 0
BALANCE_NEGATIVE  – declined because balance is negative (overpaid)
SAMEDAY_TRANSFERS – same-day sale + buyback, both cancelled
USER_REQUESTED    – explicit user cancellation
UNSOLD            – buyback requested but loan was never actually active
```

#### `ExternalAssetOwnerTransferDetails`
Snapshot of the loan's financials at the moment of transfer:
- `totalPrincipalOutstanding`
- `totalInterestOutstanding` (calculated via `ExternalAssetOwnerTransferOutstandingInterestCalculation`)
- `totalFeeChargesOutstanding`
- `totalPenaltyChargesOutstanding`
- `totalOverpaid`

#### `ExternalAssetOwnerTransferLoanMapping` (table: active loan→owner mapping)
Tracks which transfer record is currently "active" for a given loan. Deleted on buyback or owner change.

### Transfer Lifecycle (COB Step: `LoanAccountOwnerTransferBusinessStep`)
This COB business step runs with enum name `EXTERNAL_ASSET_OWNER_TRANSFER`. On each COB date it:
1. Queries for transfers with `settlementDate = today` and status in `{PENDING, PENDING_INTERMEDIATE, BUYBACK, BUYBACK_INTERMEDIATE}`.
2. **Size = 1, PENDING/PENDING_INTERMEDIATE**: Calls `sellAssetOrDecline()`.
   - Checks `LoanTransferabilityService.isTransferable()`. If not transferable, creates a DECLINED record.
   - Otherwise: expires the existing PENDING, creates a new ACTIVE (or ACTIVE_INTERMEDIATE) record, posts journal entries via `LoanJournalEntryPoster`, creates loan mapping.
3. **Size = 1, BUYBACK/BUYBACK_INTERMEDIATE**: Calls `handleBuyback()`.
   - Looks for an existing ACTIVE record for the same owner.
   - If found: expires both ACTIVE and BUYBACK, posts reverse journal entries.
   - If not found (loan was PENDING but not yet sold): creates CANCELLED/UNSOLD.
4. **Size = 2 (same-day sale + buyback)**: Both transfers are cancelled with sub-status `SAMEDAY_TRANSFERS`.

### Delayed Settlement (Intermediary Path)
When `DelayedSettlementAttributeService` detects that a loan product has the delayed-settlement attribute enabled:
- A loan is first sold to an **intermediary** (`PENDING_INTERMEDIATE` → `ACTIVE_INTERMEDIATE`).
- Later, a second sale moves ownership from the intermediary to the **ultimate investor** (`PENDING` → `ACTIVE`, with the intermediary recorded as `previousOwner`).
- Intermediary sale + direct sale cannot coexist on the same day (throws `IllegalStateException`).

### Loan Product Attributes
API: `GET/POST/PUT /v1/external-asset-owners/loan-product/{loanProductId}/attributes`

Allows configuring per-product investor attributes (e.g., the `SETTLEMENT_MODEL` attribute key that controls delayed-settlement behavior).

Entity: `ExternalAssetOwnerLoanProductAttributes` (table: `m_external_asset_owner_loan_product_attributes`)

### REST API (`ExternalAssetOwnersApiResource`)
Base path: `/v1/external-asset-owners`

| Method | Path | Command | Description |
|--------|------|---------|-------------|
| POST | `/transfers/loans/{loanId}?command=sale` | `sale` | Initiate loan sale by loan ID |

### Event System
- `LoanOwnershipTransferBusinessEvent` – fired after each sale/buyback/cancel during COB.
- `LoanAccountSnapshotBusinessEvent` – fired after successful sale or buyback.

### Data Enrichers
The module provides JAX-RS response enrichers that add `ownerExternalId` and transfer context to:
- `LoanAccountDataV1Enricher` – enriches loan detail responses
- `LoanTransactionDataV1Enricher` – enriches loan transaction responses
- `LoanChargeDataV1Enricher` – enriches loan charge responses
- `LoanTransactionAdjustmentDataV1Enricher` – enriches adjustment responses

### Accounting
`AccountingServiceImpl` posts journal entries to the configured GL accounts for:
- Transfer-in (sale): credit the loan receivable on the MFI's books, debit transfer proceeds
- Transfer-out (buyback): reverse the above
- Investor-side entries handled via `InvestorAccountingHelper`

## 2. `fineract-working-capital-loan` — Working Capital Products

### Purpose
A specialized loan type for **working capital financing** (revolving or short-term business credit). It has its own domain model, COB pipeline, API surface, and accounting processor, separate from the standard loan module.

### Key Domain Classes

#### `WorkingCapitalLoan` (table: `m_wc_loan`)
Core entity. Fields include:
- `accountNumber`, `externalId`, `loanStatus` (reuses `LoanStatus` enum)
- `client` → `Client`
- `fund` → `Fund`
- `loanProduct` → `WorkingCapitalLoanProduct`
- `loanProductRelatedDetails` → `WorkingCapitalLoanProductRelatedDetails` (embedded, includes `discount`, `breach`, `nearBreach` config)
- `disbursementDetails` → list of `WorkingCapitalLoanDisbursementDetails`
- `balance` → `WorkingCapitalLoanBalance` (tracks `realizedIncomeFromDiscountFee`)
- `loanCounter`, `loanProductCounter` – per-client and per-product-per-client sequence numbers
- `lastClosedBusinessDate` – last COB date processed
- Optimistic locking via `@Version int version`

#### `WorkingCapitalLoanProductRelatedDetails` (embedded)
Contains product configuration embedded in each loan record: `discount` amount, `breach` configuration, `nearBreach` thresholds.

### Loan Lifecycle Commands (from `WorkingCapitalLoanConstants`)
| Command String | Description |
|---|---|
| `approve` | Approve the loan application |
| `disburse` | Disburse funds to client |
| `repayment` | Record a repayment |
| `goodwillCredit` | Apply a goodwill credit |
| `creditBalanceRefund` | Refund excess credit balance |
| `discountFee` | Apply a discount fee |
| `discountFeeAdjustment` | Adjust a prior discount fee |
| `undo` | Undo last action (approval or disbursal) |

### API Surface

| Endpoint | Operation |

| `POST /{loanId}?command=repayment` | Record repayment |

Additional sub-resources:
- `/v1/working-capital-loans/{loanId}/transactions` – transaction history (via `WorkingCapitalLoanTransactionsApiResource`)
- `/v1/working-capital-loans/{loanId}/charges` – loan charges (via `WorkingCapitalLoanChargesApiResource`)

- `/v1/working-capital-loans/{loanId}/breach-schedule` – breach schedule data
- `/v1/working-capital-loans/{loanId}/delinquency-range-schedule` – delinquency range schedule
- `/v1/working-capital-loans/{loanId}/amortization-schedule` – projected amortization schedule

#### Amortization Schedule Calculator
`DefaultProjectedAmortizationScheduleCalculator` computes a projected payment schedule using time-value-of-money functions (`TvmFunctions`). Takes `totalPaymentVolume` and `principalAmount` as key inputs.

### COB Pipeline for Working Capital Loans

The WC COB runs independently from the standard loan COB. It has its own:
- `WorkingCapitalLoanCOBManagerConfiguration` – Spring Batch job configuration
- `WorkingCapitalLoanCOBWorkerConfiguration` – worker step configuration
- `WorkingCapitalLoanCOBPartitioner` – partitions WC loans across workers
- `WorkingCapitalLoanAccountLock` (table: `m_wc_loan_account_lock`) – prevents concurrent processing

#### COB Business Steps (all extend `WorkingCapitalLoanCOBBusinessStep`)

| Step Enum Name | Class | Description |
|---|---|---|
| `WC_BREACH_SCHEDULE` | `BreachScheduleBusinessStep` | Generates/extends breach schedule periods; evaluates breach on each COB date |
| `WC_NEAR_BREACH_EVALUATION` | `NearBreachEvaluationBusinessStep` | Evaluates whether a loan is approaching a breach threshold |
| `WC_DISCOUNT_FEE_AMORTIZATION` | `DiscountFeeAmortizationBusinessStep` | Amortizes discount fees (recognized income from discount) daily via COB |
| `WC_DELINQUENCY_CLASSIFICATION` | `WorkingCapitalLoanDelinquencyClassificationBusinessStep` | Assigns delinquency bucket tags to WC loans |
| `WC_DELINQUENCY_RANGE_SCHEDULE` | `DelinquencyRangeScheduleBusinessStep` | Maintains the delinquency range schedule |
| `DUMMY` | `DummyBusinessStep` | No-op placeholder for testing |

### Business Events
Fired on WC loan transactions:
- `WorkingCapitalLoanDisbursalTransactionBusinessEvent`
- `WorkingCapitalLoanRepaymentTransactionBusinessEvent`
- `WorkingCapitalLoanDiscountFeeTransactionBusinessEvent`
- `WorkingCapitalLoanDiscountFeeAdjustmentTransactionBusinessEvent`
- `WorkingCapitalLoanCreditBalanceRefundTransactionBusinessEvent`
- `WorkingCapitalLoanUndoDisbursalTransactionBusinessEvent`

### Accounting
`WorkingCapitalLoanAccountingProcessor` handles journal entry generation for WC loans, analogous to the standard loan accounting processor but tailored for WC-specific transaction types (discount fees, etc.).

## 3. `fineract-mix` — MIX Market XBRL Reporting

### Purpose
Integration with the **MIX Market** (Microfinance Information eXchange) reporting standard. Generates XBRL (eXtensible Business Reporting Language) reports that MFIs can submit to MIX Market for benchmarking and transparency.

### Key Components

#### REST API
| Path | Method | Description |
|---|---|---|
| `GET /v1/mixreport` | GET | Generate and return an XBRL report as `application/xml`. Query params: `startDate`, `endDate`, `currency`. |
| `GET /v1/mixtaxonomy` | GET | List all MIX taxonomy entries (the XBRL concept definitions). |
| `GET /v1/mixmapping` | GET | Retrieve the current taxonomy → GL account mapping. |
| `PUT /v1/mixmapping` | PUT | Update the taxonomy mapping (maps MIX XBRL concepts to Fineract GL accounts). |

#### Domain Model
- **`MixTaxonomy`** (table: `mix_taxonomy`) – An XBRL concept/element. Fields: `name`, `namespaceId`, `dimension`, `type`, `description`.
- **`MixTaxonomyMapping`** (table: `mix_xbrl_mapping`) – Maps a `MixTaxonomy` entry to a GL account query result.
- **`MixReportXBRLNamespace`** (table: `mix_xbrl_namespace`) – Namespace declarations needed in the XBRL document (e.g., `xmlns:gl-cor="..."` ).

#### Report Generation
1. `MixReportXBRLResultService` queries Fineract's financial data (keyed by GL mappings) for the given date range and currency.
2. `MixReportXBRLBuilder` assembles the result into well-formed XBRL XML, injecting the configured namespaces and context (period start/end dates, entity identifier).
3. The mapping between MIX taxonomy elements and actual GL account balances is configured via the `/v1/mixmapping` API.

## 4. `fineract-branch` — Branch Cash Management (Tellers & Cashiers)

### Purpose
Branch-level **teller cash management**. Allows an organization to assign tellers to offices and cashiers to tellers, then record cash allocations and settlements. The module is in package `org.apache.fineract.organisation.teller`.

### Key Domain Classes

#### `Teller` (table: `m_tellers`)
Represents a physical or logical cash point at an office. Fields:
- `office` → `Office` (the branch)
- `debitAccount` / `creditAccount` → `GLAccount` (accounting entries for cash in/out)
- `name`, `description`
- `startDate`, `endDate` – operational validity period
- `status` (integer, maps to `TellerStatus` enum: `ACTIVE`, `INACTIVE`, `CLOSED`)
- `cashiers` – set of `Cashier` entities

#### `Cashier` (table: `m_cashiers`)
A staff member assigned to a teller. Tracks:
- `teller`, `staff`
- `description`, `startDate`, `endDate`
- `isFullDay` flag (whether the cashier works the full day)
- `startTime`, `endTime`

#### `CashierTransaction` (table: `m_cashier_transactions`)
A cash movement record. Fields: `cashier`, `txnType` (from `CashierTxnType`), `txnAmount`, `txnNote`, `txnDate`, `entityType`, `entityId`.

#### `CashierTxnType`
```
101 – Allocate Cash   (from branch safe to cashier drawer)
102 – Settle Cash     (from cashier drawer back to branch safe)
103 – Cash In         (incoming transaction, e.g., loan repayment received)
104 – Cash Out        (outgoing transaction, e.g., loan disbursement)
```

#### `TellerJournal`
Summary view of cashier transactions grouped by teller/date. Used for end-of-day reconciliation.

## 5. `fineract-charge` — Charge Definitions

### Purpose
Defines the **charge catalog** — the reusable fee/penalty templates that can be attached to loan products, savings products, clients, or share accounts.

### Key Domain Classes

#### `Charge` (table: `m_charge`)
The master charge definition. Fields:
- `name` – unique name
- `amount` – default amount
- `currencyCode`
- `chargeAppliesTo` → `ChargeAppliesTo` (see below)
- `chargeTimeType` – when the charge is due (see `ChargeTimeType`)
- `chargeCalculation` → `ChargeCalculationType`
- `chargePaymentMode` → `ChargePaymentMode`
- `feeOnDay`, `feeOnMonth` – for annual fees on a specific date
- `feeInterval` – recurring fee interval
- `penalty` – boolean, distinguishes fees from penalties
- `active` – whether the charge is usable
- `minCap`, `maxCap` – optional floor/ceiling on percentage-based charges
- `taxGroup` → `TaxGroup` – optional tax applicability

#### `ChargeAppliesTo` (enum)
```
LOAN (1)
SAVINGS (2)
CLIENT (3)
SHARES (4)
WORKING_CAPITAL_LOAN (5)
```

#### `ChargeCalculationType` (enum)
| Value | Code | Description |
|---|---|---|
| 1 | `FLAT` | Fixed amount |
| 2 | `PERCENT_OF_AMOUNT` | % of outstanding principal |
| 3 | `PERCENT_OF_AMOUNT_AND_INTEREST` | % of principal + interest |
| 4 | `PERCENT_OF_INTEREST` | % of interest only |
| 5 | `PERCENT_OF_DISBURSEMENT_AMOUNT` | % of disbursement amount |

For **Working Capital Loans**, only `FLAT` is valid.

#### `ChargePaymentMode` (enum)
```
REGULAR (0)        – charged directly
ACCOUNT_TRANSFER (1) – auto-deducted via linked savings account
```

| Operation | Endpoint |
|---|---|

## 6. `fineract-tax` — Tax Groups and Components

### Purpose
Manages **tax configuration** that can be applied to charges. Tax is computed on top of charges when the charge's `taxGroup` is set.

### Key Domain Classes

#### `TaxComponent` (table: `m_tax_component`)
A single tax rate. Fields:
- `name`
- `percentage` (e.g., `18.000000` for 18% GST)
- `debitAccountType`, `debitAccount` → `GLAccount` (where tax debit is posted)
- `creditAccountType`, `creditAccount` → `GLAccount` (where tax credit is posted)
- `startDate` – effective from date
- `taxComponentHistories` → set of `TaxComponentHistory` (rate change audit trail)
- `taxGroupMappings` → set of `TaxGroupMappings`

#### `TaxComponentHistory`
Stores historical percentage values and date ranges for a `TaxComponent`. Allows rate changes over time without losing historical data.

#### `TaxGroup` (table: `m_tax_group`)
A named collection of `TaxComponent` entries. Fields:
- `name`
- `taxGroupMappings` → set of `TaxGroupMappings`

#### `TaxGroupMappings` (table: `m_tax_group_mappings`)
Join table linking `TaxGroup` to `TaxComponent` with effective date ranges.

| `PUT /v1/taxes/component/{taxComponentId}` | PUT | Update a tax component |

`ChargeTaxApplicationService` computes the tax due on a charge based on the charge amount and the applicable `TaxGroup`. Tax is applied at the time of charge creation/calculation and posted as separate journal entries to the configured debit/credit GL accounts.

## 7. `fineract-rates` — Floating Interest Rates

### Purpose
Manages **floating (variable) interest rate** definitions. Loan products can be linked to a floating rate so that the effective interest rate is determined dynamically by the floating rate in effect on each calculation date, rather than a fixed rate on the product.

### Key Domain Classes

#### `FloatingRate` (table: `m_floating_rates`)
A named floating rate. Fields:
- `name` (unique)
- `isBaseLendingRate` – if `true`, this rate is the base lending rate (e.g., a benchmark like LIBOR/SOFR/MCLR)
- `isActive`
- `floatingRatePeriods` → ordered list of `FloatingRatePeriod` (by `fromDate ASC, id ASC`)

#### `FloatingRatePeriod` (table: `m_floating_rates_periods`)
One rate period within a floating rate. Fields:
- `floatingRate` – parent
- `fromDate` – effective start date of this rate
- `interestRate` – the rate value (e.g., `8.500000` for 8.5% p.a.)
- `isDifferentialToBaseLendingRate` – if `true`, the `interestRate` is a spread added to the current base lending rate
- `isActive`

#### `FloatingRateDTO`
Transfer object combining rate details. Used by the loan engine to look up the applicable rate for a given date.

#### `InterestRatePeriodData`
Used in loan schedule output to show the floating rate history that contributed to the computed interest on each period.

### Rate Lookup Logic
When computing loan repayment schedules:
1. The loan's product is checked for a linked `FloatingRate`.
2. The `FloatingRatePeriod` with `fromDate <= calculationDate` and the highest `fromDate` (most recent) is selected.
3. If `isDifferentialToBaseLendingRate = true`, the current base lending rate's value is fetched and added.
4. The resulting rate is applied to the period's principal balance.

### REST API (`FloatingRatesApiResource`)
Base path: `/v1/floatingrates`

| Operation | Endpoint |
|---|---|
| Create floating rate | `POST /v1/floatingrates` (mandatory: `name`; optional: `isBaseLendingRate`, `isActive`, `ratePeriods`) |
| List all floating rates | `GET /v1/floatingrates` |
| Get floating rate | `GET /v1/floatingrates/{floatingRateId}` |
| Update floating rate | `PUT /v1/floatingrates/{floatingRateId}` (note: past periods cannot be changed; future periods are replaced) |
